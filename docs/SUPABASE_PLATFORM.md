This document summarizes the conversion of Healthcare Copilot into a Supabase-backed
healthcare analytics platform with a realistic, large-scale synthetic dataset.

## Schema summary
19 tables — **12 clinical/reference** on the copilot SQL allow-list, **7
application/system** (auth/audit/copilot/alerts) excluded from AI access. UUID
PKs, FK indexes, GUC-driven RLS (`ENABLE`+`FORCE`), least-privilege `hc_readonly`
executor, Fernet-encrypted PHI on `users`. Full ERD in
[DATABASE_SCHEMA.md](DATABASE_SCHEMA.md).

## Dataset realism
US-representative demographics; prevalence-weighted, age-gated ICD-10 diagnoses;
cross-table correlation (diabetic → high glucose/HbA1c + metformin/insulin);
encounter-type intensity; payer/charge/claim financials; in-database 30-day
readmission derivation. Details: [DATA_GENERATION.md](DATA_GENERATION.md).

## Storage estimates
Measured: `small` (~306K rows) = **90 MB** incl. indexes on PostgreSQL 16.
Extrapolated: `medium` (~3.5M) ≈ 1.2–1.6 GB; **`prod` (~6.6M) ≈ 2.5–4 GB**.
Comfortably within Supabase Pro disk; the included `PHI_ENCRYPTION_KEYS` live in a
separate secret manager.

## Performance (measured, `small` scale, local PG16, warm cache)
8 representative BI/copilot queries, **p50 ≈ 50–69 ms, p95 ≈ 50–71 ms** over
~220K core fact rows. The ~50 ms floor is local round-trip (a bare `COUNT(*)` is
also 50 ms). Full table: [PERFORMANCE_REPORT.md](PERFORMANCE_REPORT.md). At `prod`
scale, the FK/`admit_date`/`icd10_code`/`loinc_code` indexes keep the cohort and
trend queries index-bound.

## What was verified
End-to-end against real PostgreSQL in an isolated schema: `supabase/migration.sql`
applies cleanly (tables + indexes + RLS + defaults); the pipeline COPY-loads every
table at demo and small scale; `validate_data.py` passes (0 FK orphans, 0
unexpected NULLs, correct physician/nurse split, all analytics scenarios return
rows); `benchmark.py` produces real numbers; single Alembic head `d4e5f6a7b8c9`;
app modules import cleanly.

## Remaining limitations (honest)
- **Full `prod` (6.6M) load was not executed here** — no Supabase project /
  credentials in this environment. The pipeline is validated at demo + small
  scale and is scale-only-different at `prod`; run it against your project with
  `--scale prod`. Seed from a same-region VM for best throughput.
- **Child-table totals are approximate** (Poisson-rated to hit targets in
  expectation, ±~2%), not exact — appropriate for synthetic analytics.
- **Per-entity standalone child scripts** (`seed_diagnoses`/`lab_results`/`claims`)
  don't apply cross-table condition bias; only the full `seed_all` pipeline does.
- **Supabase Auth/Storage/Realtime** are not used — the app keeps its own JWT auth
  and connects directly to Postgres. `anon`/`authenticated` are REVOKE'd from PHI
  tables; disable the PostgREST data API entirely if unused.
- **Synthetic data only** — no real PHI. Sign a Supabase BAA before real PHI.
